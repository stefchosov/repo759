1a. The grouping is based on the computational complexity of the operation relative to the
input size. Level 1 functions operate on vectors only (e.g. dot products, norms, axpy),
so their complexity is O(n) in both work and memory. Level 2 functions involve a
matrix and a vector (e.g. gemv), giving O(n^2) work. Level 3 functions operate on two
or more matrices (e.g. gemm), giving O(n^3) work for O(n^2) memory. The higher the
level, the better the compute-to-memory ratio, which is why Level 3 operations are
easiest to optimize on hardware like GPUs that are memory bandwidth limited.

1b. Two examples of structure-specialized cuBLAS functions:

  - cublasSsymm: performs matrix-matrix multiplication where one of the input
    matrices is symmetric. Because A = A^T, only the upper or lower triangle needs
    to be stored and read, cutting memory traffic roughly in half versus a general
    gemm while producing the same result.

  - cublasStrmm: multiplies a general matrix by a triangular matrix. Since a
    triangular matrix is zero above or below the diagonal, roughly half the
    multiplications are trivially zero and can be skipped. This reduces both work
    and memory accesses compared to treating it as a general matrix.

1f. cuBLAS is significantly faster than our HW05 tiled implementation across all problem
sizes tested (n = 32 to 2048). At n = 2048 our tiled matmul with block_dim=16 took
around 23 ms for float. cuBLAS at similar sizes is well under 1 ms. The gap grows
with n because cuBLAS uses architecture-specific optimizations that our implementation
simply cannot match: it auto-tunes tile sizes at runtime, uses tensor cores for fused
multiply-add operations (which our kernel does not), and employs register-level tiling
to further hide latency. Our tiled implementation reduces global memory traffic by
a factor of block_dim compared to the naive approach, but cuBLAS squeezes much more
parallelism out of the hardware by better utilizing warp-level operations and on-chip
memory hierarchy. Essentially our implementation is a textbook tiled GEMM while
cuBLAS is a years-refined production library—so the performance difference is expected
and not surprising.
