Task 4 Analysis — GPU Algorithm Comparison and Memory-Hierarchy Effects
========================================================================

Goal
----
Compare two fundamentally different GPU collision-search algorithms against
Task 3's Pollard's rho:

  Algorithm A (Task 3): Pollard's rho with distinguished points.
    Memory: O(1) per thread. 8 MB total device buffer regardless of bit width.
    Deterministic single launch — total time is wall time.

  Algorithm B (Task 4): Thrust parallel sort.
    Memory: O(N) where N ≈ multiplier × sqrt(π/2) × 2^(bits/2).
    Three phases per round: parallel hash → thrust::sort_by_key → parallel
    duplicate-scan kernel. Each round has find rate p = 1 − e^(−mult), so
    average rounds = 1/p.

Algorithm B is run in two memory modes:

  Device mode (cudaMalloc): pure GPU VRAM. Capped at bits=48 by 8 GB GPU.
  Unified mode (cudaMallocManaged): pages migrate between GPU/host on demand.
    Lets the sort exceed VRAM and spill into 96 GB host RAM at the cost of
    PCIe-bound performance during the sort.

Adaptive allocation multiplier:
  bits ≤ 48 : 4× (one-shot, ~99% find rate)
  bits = 52 : 2× (~63% find rate per round, avg 1.6 rounds)
  bits = 56 : 1× (~63% find rate per round, avg 1.6 rounds — but variance
                  is high since geometric distribution has stdev = mean for p≈0.6)


Finding 1 — Unified Memory Has Real but Variable Overhead (1.6×–4×)
--------------------------------------------------------------------
Comparing unified vs device mode at bit widths where the working set fits
entirely in 8 GB VRAM (no page migration expected):

  bits   md5 dev/uni   sha1 dev/uni   sha256 dev/uni
  ----   -----------   ------------   --------------
   24      3.67×          3.98×          3.72×
   32      2.25×          0.64× *        0.58× *
   40      2.17×          1.64×          1.79×
   48      2.02×          2.25×          2.37×

  (* SHA-1 and SHA-256 at bits=32 are sub-millisecond and dominated by
   measurement noise. Ignore.)

The trend is consistent for MD5 (~2× steady) and at bits=48 across all
algorithms (2.0–2.4×). At small bits the times are sub-millisecond and
ratios fluctuate widely. Conclusion: the page-tracking driver tax is real
— roughly 1.6×–2.4× steady at meaningful problem sizes — but the absolute
time difference is modest. On a log-scale plot (5 orders of magnitude),
this gap is visually compressed and easy to miss; the linear-scale
overhead-ratio plot (task4_overhead.pdf) makes it clear.


Finding 2 — At bits=56, Thrust Unified is Competitive with Pollard's ρ;
                Total Time is Dominated by Retry Variance
-----------------------------------------------------------------------
Initial framing was wrong. Per-round measurements at bits=56:

  algo     Pollard ρ (ms)   Thrust uni total (ms)   rounds   per round (ms)
  ----     --------------   ----------------------   ------   --------------
  md5            2,133              3,359               6.0          560
  sha1           2,375              1,161               1.0        1,161
  sha256         1,930              1,951               3.0          650

Per-round Thrust unified time is FASTER than Pollard's rho's deterministic
single-launch time for all three algorithms (560–1,161 ms vs Pollard's
1,930–2,375 ms). The bits=56 results vary wildly because I used 1×
over-allocation, which gives only ~63% find rate per round — total time
is dominated by retry luck:

  - md5:    very unlucky (6 rounds), total 1.57× SLOWER than Pollard
  - sha1:   very lucky  (1 round),  total 2.04× FASTER than Pollard
  - sha256: average     (3 rounds), total ~1.0× tied with Pollard

The honest conclusion: at bits=56 (and presumably above), Thrust unified
and Pollard's rho are in the same performance class. Pollard's rho's
advantage is determinism, not raw speed. Thrust unified's variance comes
from the geometric retry distribution, not from the sort itself. Higher
allocation multipliers (e.g., 2× at bits=56) would reduce variance but
require >16 GB VRAM working set — which is precisely the spill scenario
unified memory is intended to solve.


Finding 3 — PCIe Spill Is Sharp at the VRAM Boundary
-----------------------------------------------------
Tracking unified-mode time as bits cross the 8 GB VRAM cap (md5):

  bits   md5 unified (ms)   factor vs prior   theoretical 2^(bits/2)×
  ----   ----------------   ----------------   ------------------------
   48           240.9              ―                  ―
   52           516.4              2.14×             2.0×    (matches: still fits)
   56         3,359.1              6.50×             4.0×    (extra 1.6× = spill)

From 48 → 52 the slowdown matches theory (2.14× vs 2.0× expected) — the
sort working set still fits in VRAM. From 52 → 56 the slowdown is 6.5×
where theory predicts only 4×. The extra 1.6× factor IS the PCIe-bound
sort. (Note: this 1.6× is also amplified by the 6-round retry; per-round
the factor is 4.65×/4 = 1.16× extra, still real but more modest.)

The crossover is sharp because radix sort scratch space allocations are
discrete — once the working set exceeds VRAM by even a small amount, the
GPU's L2 + VRAM caches stop containing the whole pass and every memory
access has a chance of triggering a remote-fault PCIe migration.


GPU Algorithm Selection Guide (based on measured evidence)
-----------------------------------------------------------
  Bit width     Best algorithm                Reason
  -----------   --------------------------    --------------------------------
  bits ≤ 48     Thrust sort (device mode)     Fastest absolute (5–35× faster
                                                than Pollard's rho) thanks to
                                                cache-friendly radix sort with
                                                no chain-walk overhead.
  bits 52       Thrust sort (either mode)     Both modes work; device a bit
                                                tighter on VRAM, unified pays
                                                ~2× page-tracking tax but
                                                still beats Pollard's rho.
  bits = 56     Comparable, choose by         Per-round Thrust unified beats
                preference                      Pollard's rho. Total time
                                                depends on retry luck.
                                                Pollard's rho is preferred for
                                                deterministic time budgets.
  bits = 64     Pollard's rho only            Thrust unified at bits=64 was
                                                attempted with 0.5× allocation
                                                (~34 GB working set). Every
                                                trial crashed inside
                                                thrust::sort_by_key with a
                                                temp-buffer allocation
                                                failure. Even with 96 GB host
                                                RAM available via Unified
                                                Memory, Thrust's internal
                                                radix-sort scratch cannot be
                                                satisfied at this batch size.
                                                Pollard's rho runs cleanly
                                                in 8–10 sec at bits=64.
  bits ≥ 72     Pollard's rho only            Working set exceeds 96 GB host
                                                RAM even at low allocation
                                                factors.


Methodology Notes
-----------------
- Trial counts: 10 trials for both device and unified modes. Device mode
  has small but nonzero variance (1.8% miss rate at 4× allocation, plus
  GPU clock-boost / kernel-launch jitter at the ~5–10% level). Unified
  mode has much larger variance from the geometric retry distribution
  at sub-1× allocation. Reported numbers are median; uncertainty bands
  are the inter-quartile range (IQR). Device-mode runs are sub-second
  per trial so 10× repetition is essentially free.
- bits=16 device-mode MD5 row (218 ms) is GPU/CUDA initialization on the
  very first kernel launch of the binary — ignored in plots and analysis.
- Per-round time is computed as total_ms ÷ (count ÷ expected). This treats
  the count column as a proxy for "rounds × expected per round."
- bits=64 in unified mode was attempted with 0.5× expected over-allocation
  (~34 GB working set per round). Every invocation died inside
  thrust::sort_by_key with a temp-buffer allocation failure — Thrust's
  internal radix-sort scratch space at this batch size could not be
  satisfied even with 96 GB host RAM available. I added try/catch around
  the sort call so the binary marks the row as -1 and continues to the
  next configuration; bits=64 was subsequently dropped from the unified
  bits sweep. Pollard's rho is the only working approach at bits >= 64.
