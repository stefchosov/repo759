Task 3 — Truncated-Collision Search: CPU Serial vs OpenMP vs GPU (Pollard's rho)
================================================================================

Overview
--------
This task searches for collisions in truncated versions of MD5, SHA-1, and SHA-256.
A "truncated collision" means two distinct inputs whose hash outputs agree in the
lowest N bits. We sweep N in {16, 24, 32, 40, 48, 56, 64} and compare three
implementations: CPU serial, CPU parallel (OpenMP), and GPU (Pollard's rho).

Birthday Paradox Foundation
----------------------------
By the birthday paradox the expected number of random inputs needed to find a
collision in an N-bit hash space is:

    E[collisions] ≈ sqrt(π/2) × 2^(N/2)

This grows as 2^(N/2), so each additional 8 bits doubles the required work twice
(4× harder).

The CPU serial implementation validates this directly: cpu_count is the number
of distinct inputs evaluated before the first collision is found. Across all
algorithms and bit widths, cpu_count / expected is consistently close to 1.0,
confirming that the truncated hash extraction is correct and the output
distribution is sufficiently uniform for the birthday paradox to hold.

The gpu_count column is NOT comparable to expected. It measures total hash
iterations across all 65,536 Pollard's rho threads across all kernel launches —
every thread walks its full chain length regardless of when a collision is
detected. At bits=32 for example, a single launch computes 65,536 × 1,024 =
67M hash evaluations while expected is only ~82K. gpu_count is therefore orders
of magnitude larger than expected by design; it is useful only for computing
GPU throughput (hashes/second), not for birthday paradox validation. The
meaningful performance comparison between CPU and GPU is time (cpu_ms vs
gpu_ms), not count.

GPU Algorithm — Pollard's Rho with Distinguished Points
---------------------------------------------------------
The GPU runs Pollard's rho with distinguished points rather than a naive birthday
search. Each of 65,536 CUDA threads walks an independent Markov chain:

    x_{n+1} = truncate(hash(x_n), N bits)

A "distinguished point" (DP) is any chain value x whose low dp_bits = N/4 bits
are all zero. When two threads reach the same DP from different starting values,
a collision exists between their chains. This approach uses O(1) memory per
thread — the GPU only stores the DP buffer (8 MB), so it scales to bits=64
without hitting device memory limits.

Template kernels (template<int ALGO>) eliminate warp divergence: the compiler
emits three separate specializations for MD5, SHA-1, and SHA-256, with no
runtime branches inside the hot loop.

CPU Serial Algorithm
--------------------
Sequential scan of inputs 0, 1, 2, ... hashing each and inserting into an
std::unordered_map<uint64_t, uint64_t>. The first duplicate key is a collision.
Run for bits ≤ 56; bits=56 requires ~420M map entries (~15–17 GB with node
overhead) and is accommodated by requesting 96 GB SLURM memory. bits=64 is not
feasible on any available CPU and is reported as -1, running only on GPU.

OpenMP Parallel Algorithm — Thread Count Choice
------------------------------------------------
The OpenMP implementation uses a two-phase approach per batch:

  Phase 1 (parallel): N threads each hash their slice of the current index range
    into pre-allocated hbuf/ibuf arrays. No synchronization — threads write to
    disjoint array positions.

  Phase 2 (serial): merge the batch into a global std::unordered_map, stopping
    at the first duplicate key.

A dedicated thread-count scaling experiment was run on Euler sweeping
OMP_NUM_THREADS in {1, 2, 4, 8, 12, 16, 20, 24, 28, 32} at bits={40, 48}.
Performance peaked at 12 threads and degraded beyond that. Three factors explain
this sweet spot:

  1. NUMA boundary (primary cause).
     Euler's compute nodes are dual-socket. Each socket has its own memory
     controller. The first 12 cores all reside on socket 0 and access local
     DRAM at full bandwidth. Threads 13+ cross to socket 1, incurring remote
     memory latency (~2× slower). The hash buffers (hbuf/ibuf, ~32 MB at
     bits=48) are allocated on socket 0's memory, so every cross-socket
     thread pays a NUMA penalty on every read and write.

  2. Amdahl's Law — serial merge bottleneck.
     Phase 2 is entirely serial. As thread count rises, Phase 1 shrinks but
     Phase 2 is constant. Past ~12 threads Phase 2 becomes the dominant term,
     so additional threads yield diminishing returns before NUMA even matters.

  3. L3 cache pressure.
     The batch working set at bits=48 with 12 threads is ~32 MB, which sits
     near the per-socket L3 cache capacity. Beyond 12 threads the larger batch
     spills to DRAM, adding memory latency on top of the NUMA effect.

All three scaling sbatch scripts therefore fix OMP_NUM_THREADS=12 and request
--cpus-per-task=12 to stay on a single socket and avoid NUMA penalties.

Why bits=64 Is CPU-Infeasible (And What It Would Take)
-------------------------------------------------------
bits=64 expects ~2.7 billion entries before a collision. The current
unordered_map implementation requires:

  Memory:  ~100–110 GB (bucket array ~22 GB + node storage ~65 GB)
  Time:    hashing ~1 second (12 threads × 200M MD5/s), but the serial
           merge phase performs 2.7B random memory accesses across 100 GB —
           at ~100 ns DRAM latency each that is ~270 seconds of map ops alone.
           Total wall time: ~5 minutes even with sufficient memory.

Why OpenMP Does Not Help at bits=64
-------------------------------------
OpenMP parallelizes Phase 1 (hashing) only — Phase 2 (map insertion) remains
entirely serial. At bits=64 the map itself requires ~100 GB regardless of how
many threads are hashing. More threads in Phase 1 cannot reduce the memory
footprint or the serial merge cost; the bottleneck shifts from compute to
memory capacity and DRAM latency. CPU_MAX_BITS=56 therefore gates both the
serial and OpenMP implementations equally — they share the same unordered_map
and the same memory constraint.

A hypothetical optimized CPU implementation (parallel sort-based) would reduce
both constraints significantly:

  1. Replace unordered_map with a flat (hash, index) pair array.
     Memory drops to ~43 GB (2.7B × 16 bytes) — fits on a high-memory server.

  2. Parallel radix sort the array after Phase 1 hashing.
     Phase 1 (hashing):  ~1 s  (12 threads, embarrassingly parallel)
     Phase 2 (sort):     ~10–20 s  (parallel radix sort, cache-friendly)
     Phase 3 (scan):     ~1 s  (linear scan for adjacent duplicates)
     Total:              ~20–30 s on a 12-core machine with 64 GB RAM.

Even with this optimization, the GPU Pollard's rho completes bits=64 in
single-digit seconds using only 8 MB of device memory. The GPU advantage at
high bit counts is therefore algorithmic, not just clock speed: Pollard's rho
eliminates memory scaling entirely. CPU approaches, no matter how well
parallelized, must touch O(2^(N/2)) memory to detect collisions via the
birthday paradox, making them memory-bandwidth-bound at scale.

GPU vs CPU Speedup
------------------
The GPU outperforms both CPU implementations for all bit widths ≥ 32. The
speedup grows with bits because:
  - CPU serial time grows as 2^(N/2) with no additional parallelism
  - OpenMP speedup is capped by the serial merge phase (Amdahl) and NUMA
  - The GPU runs 65,536 Pollard chains simultaneously with O(1) memory each,
    so its effective work per second scales independently of the hash space size

For bits=56 and bits=64, only the GPU produces results within the job time
and memory constraints available on Euler.

Truncated Hash Extraction
--------------------------
All three implementations extract the same 8-byte truncated value:
  MD5 (little-endian output):  first 8 digest bytes as LE uint64
    = state[0] | (state[1] << 32)
  SHA-1, SHA-256 (big-endian): first 8 digest bytes byte-swapped to LE uint64
    = bswap32(state[0]) | (bswap32(state[1]) << 32)

This ensures CPU and GPU collision counts are comparable and the birthday paradox
ratio is computed against the correct hash space.
